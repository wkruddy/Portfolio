Persistent memory for `portfolio-research`. Append a run-log row after **every** research session.

## Methodology (current)

### Discovery playbook

1. **Start internal** — `SOUL.md`, `projectData.ts`, hero/contact copy; list gaps vs Staff++ goals.
2. **Plan 3–6 queries** — Mix specificity (“principal engineer portfolio platform”) and pattern (“staff engineer site minimal IA”).
3. **Browse** — WebSearch for discovery; open `sources.md` URLs for depth (not only snippets).
4. **Score** — Each reference 1–5 on five criteria (see research skill); keep only ≥3.5 for the brief.
5. **Extract** — Patterns: IA (tabs vs scroll), hero proof type, work section honesty, motion level, contact friction.
6. **Hand off** — Update `research-to-ux.md`; reference this file’s version/date in the brief.
7. **Persist** — Append run log below; promote URLs to `sources.md` if reusable.

### What works (so far)

- Curated principal/staff sites with **narrative + scope** beat generic “15 best portfolios” listicles.
- Thematic work sections (like Kyle’s today) align with **no fake case studies** when employer work can’t be named.
- Tab-based IA (current site) matches several calm senior portfolios — don’t assume single-page scroll is required.

### What failed / avoid

- Listicles dominated by junior project grids and Webflow templates — low signal for Staff++.
- Scraping metrics or employer names from reference sites into Kyle’s brief.
- One-off searches without logging queries — repeats the same dead ends.
